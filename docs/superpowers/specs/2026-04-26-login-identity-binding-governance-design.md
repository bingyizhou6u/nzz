# 登录身份绑定治理设计方案

## 1. 背景

当前生产系统已经使用 Cloudflare Access 作为入口认证，并在 Worker 内验证 Access JWT。系统内部再用 Access email 匹配 `people.login_email`，找到启用人员和角色权限。

现有缺口是：`login_email` 已存在于数据库，但人员治理页面还不能维护它，生产首个管理员仍需要通过后台 SQL 绑定邮箱。长期看，这会让登录用户管理脱离系统审计，也容易把“业务人员”和“可登录用户”混在一起。

本阶段目标是把登录身份绑定纳入基础资料治理中心，使系统可以明确表达：

- 有 `login_email` 的人员是可登录用户。
- `login_email` 为空的人员只是业务人员，不能登录系统。
- 登录邮箱、角色、启停状态的高风险变更都受权限和审计控制。

## 2. 范围

本阶段实现：

- 人员资料维护 `login_email`。
- 人员列表展示登录邮箱、最近登录时间、登录状态。
- `login_email` 统一 trim、小写保存。
- `login_email` 可为空；为空代表不可登录。
- 只有具备 `masterData.managePeopleRoles` 的人员可以修改登录邮箱和角色。
- 普通 `masterData.write` 可维护人员名称、别名、启停状态，但不能修改登录邮箱或角色。
- 防止系统失去最后一个可登录管理员。
- 登录邮箱和角色变更写入审计日志。

本阶段不实现：

- 自建密码登录。
- 邀请邮件、重置密码、MFA 管理。
- 项目级、商户级、人员级数据范围权限。
- 多级审批和金额阈值审批。
- 独立审计中心页面。

## 3. 核心规则

### 3.1 登录身份规则

Cloudflare Access 认证通过后，系统必须按以下规则解析当前用户：

1. 读取 Access JWT 中的 email。
2. 将 email trim 并转为小写。
3. 查询 `people.login_email` 匹配且 `is_enabled = 1` 的人员。
4. 找不到人员时拒绝进入系统。
5. 人员存在但无有效角色能力时，只能得到空功能入口。

`login_email` 为空的人员不会被登录流程匹配，也不能作为当前登录 actor。

### 3.2 业务人员规则

`login_email` 为空不代表数据异常。它表示该人员只是业务对象，可继续用于：

- 经办人。
- 报销人。
- 借款人。
- 项目负责人。
- 商户负责人。
- 备用金账户所属人员。

这类人员不能登录系统，也不能作为审计 actor。

### 3.3 权限变更规则

人员字段分两组：

| 字段 | 权限 |
| --- | --- |
| `name`、`alias`、`is_enabled` | `masterData.write` |
| `roles_json`、`login_email` | `masterData.managePeopleRoles` |

如果请求试图改变角色或登录邮箱，但当前 actor 没有 `masterData.managePeopleRoles`，后端必须返回 403。

前端也要同步收敛：无权限用户看得到角色和登录状态，但不能编辑角色或登录邮箱。

### 3.4 防锁死规则

为了避免生产系统没有管理员，人员更新必须校验：

- 不允许清空最后一个启用管理员的 `login_email`。
- 不允许停用最后一个启用且绑定登录邮箱的管理员。
- 不允许移除最后一个启用且绑定登录邮箱管理员的 `admin` 角色。
- 不允许当前登录人清空自己的 `login_email` 或停用自己。

“最后一个可登录管理员”定义为：

```sql
is_enabled = 1
AND login_email IS NOT NULL
AND trim(login_email) != ''
AND roles_json includes "admin"
```

## 4. API 设计

沿用现有接口：

| 接口 | 行为 |
| --- | --- |
| `GET /api/master-data/people` | 返回人员列表，包含 `login_email`、`last_login_at`。 |
| `POST /api/master-data/people` | 创建人员，可带 `loginEmail`，但只有角色管理权限可创建可登录人员。 |
| `PATCH /api/master-data/people/:id` | 更新人员，可更新 `loginEmail`，受权限和防锁死规则约束。 |

请求字段采用前端友好的 `loginEmail`，仓储层保存为 `login_email`。

创建人员时：

- 无 `masterData.managePeopleRoles` 权限不能创建人员，因为创建人员必须分配角色。
- 有权限时可创建仅业务人员，`loginEmail` 为空。
- 有权限时可创建可登录用户，`loginEmail` 必须唯一。

更新人员时：

- 如果 `loginEmail` 未传，按空值处理，表示清空登录邮箱。
- 服务器统一做 trim、小写。
- 数据库唯一索引继续负责防重复，API 将重复邮箱错误转成明确提示。

## 5. 前端设计

人员页增加字段：

- 登录邮箱输入框。
- 登录状态列：
  - `可登录`：启用且有登录邮箱。
  - `未绑定邮箱，不可登录`：启用但无登录邮箱。
  - `已停用，不可登录`：人员停用。
- 最近登录时间列。

权限表现：

- 有 `masterData.managePeopleRoles`：可编辑角色和登录邮箱。
- 只有 `masterData.write`：可编辑姓名、别名、启停状态；角色和登录邮箱只读。
- 无 `masterData.write`：只读查看。

## 6. 审计

人员创建和更新继续写入 `audit_logs`。

当 `login_email` 或 `roles_json` 发生变化时，审计 before/after 必须包含变更后的人员快照。审计 actor 使用当前登录人员，不能使用请求体中的任意人员字段。

## 7. 错误处理

需要稳定返回以下错误语义：

| 场景 | 状态 |
| --- | --- |
| 无权限修改登录邮箱或角色 | 403 |
| 登录邮箱重复 | 400 |
| 清空最后一个可登录管理员 | 400 |
| 停用自己 | 400 |
| 清空自己的登录邮箱 | 400 |
| 登录邮箱格式明显无效 | 400 |

邮箱格式不做复杂 RFC 校验，只要求包含一个 `@`，且两侧非空，避免误拦截合法企业邮箱。

## 8. 测试计划

后端测试：

- 创建人员时保存规范化登录邮箱。
- 空登录邮箱保存为 null。
- 重复登录邮箱返回明确错误。
- 无角色管理权限不能修改登录邮箱。
- 无角色管理权限不能修改角色。
- 普通写权限可以修改名称、别名、启停状态且保留原登录邮箱和角色。
- 防止停用自己。
- 防止清空自己的登录邮箱。
- 防止移除最后一个可登录管理员。

前端测试：

- 人员表显示登录邮箱、最近登录时间、登录状态。
- 有角色管理权限时可编辑登录邮箱。
- 无角色管理权限时登录邮箱输入禁用，并在提交时保留原值。
- payload 正确使用 `loginEmail`。

## 9. 验收标准

- 管理员可以在人员页为人员绑定、修改、清空登录邮箱。
- 未绑定 `login_email` 的人员不能登录系统，但仍可作为业务人员被选择。
- 普通基础资料维护人员不能修改登录邮箱或角色。
- 系统始终保留至少一个启用、绑定邮箱、拥有 `admin` 角色的管理员。
- 权限相关测试通过，完整测试套件通过。
