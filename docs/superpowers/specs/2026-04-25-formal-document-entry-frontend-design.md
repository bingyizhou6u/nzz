# 正式单据录入前端设计

日期：2026-04-25

状态：已确认设计方向，等待实施计划。

## 1. 背景

当前前端是 MVP 骨架，能验证单据、审核、过账和报表链路，但不适合作为正式录入界面。

主要问题：

- 单据页暴露了数据库字段和 ID，项目、商户、人员、账户、科目仍可手填。
- 不同单据类型共用同一组字段，用户容易填错或漏填。
- 创建人、审核人、提交人、驳回人目前是自由文本，不符合正式台账口径。
- 主数据已经存在，但前端没有形成统一选项能力。

正式系统要求：所有真实业务数据来自结构化源单据，单据字段必须由业务规则和主数据约束，不能靠人工记 ID。

## 2. 目标

本阶段目标是把“业务单据”从工程测试表单升级为正式录入体验。

范围内：

- 建立单据录入所需的主数据选项。
- 将项目、商户、人员、账户、科目、币种、原单据全部改为选取。
- 创建人、提交人、审核人、驳回人统一从人员主数据选择。
- 按单据类型显示字段，不显示无关字段。
- 按业务关系过滤选项，例如商户跟随项目、备用金账户跟随人员。
- 缺少主数据时提示去基础资料维护，不允许手填 ID 绕过。

范围外：

- 登录、权限、角色授权。
- 附件上传。
- Excel 导入。
- 审核流多级审批。
- 大规模前端视觉重做。

## 3. 设计原则

- 用户选择业务对象，不录入数据库 ID。
- 一个字段只有一个业务含义。
- 表单跟随单据类型变化。
- 所有影响报表口径的字段必须受控。
- 可以手输的字段只限日期、金额、摘要、备注、外部凭证号等原始事实。
- 当前阶段不引入登录系统，但所有操作者字段都保存为 `people.id`。

## 4. 方案取舍

考虑过三种做法：

| 方案 | 做法 | 结论 |
| --- | --- | --- |
| A | 保留当前表单，只给 ID 输入框加校验 | 仍然要求用户知道 ID，不符合正式录入体验。 |
| B | 把所有 ID 输入框直接换成下拉 | 改动小，但不同单据仍共用同一组字段，误填风险还在。 |
| C | 建立主数据选项层，并按单据类型渲染表单 | 作为正式系统方向采用。 |

本设计采用方案 C。它能同时解决脏数据、误填字段和操作者不规范的问题，后续也能平滑接入登录权限。

## 5. 选项数据

前端需要一个单据录入选项层，来源于主数据和已审核单据。

基础选项：

| 选项 | 来源 | 用途 |
| --- | --- | --- |
| 人员 | `people` | 创建人、提交人、审核人、驳回人、经办人、报销人、借款人 |
| 项目 | `projects` | 项目收入、报销、借款核销等归属 |
| 商户 | `merchants` | 项目收入来源，费用可选归属 |
| 账户 | `accounts` | 收款、付款、换汇、备用金、还款账户 |
| 币种 | `currencies` | 单据金额币种 |
| 科目 | `categories` | 收入、费用、换汇、借款、核销分类 |
| 原单据 | `documents` | 冲正、修正、重录关联原单 |

选项只返回启用或 active 状态的数据。禁用或非 active 主数据不出现在新增单据的选项里，但历史单据仍按保存值展示。

建议接口：

- `GET /api/document-entry/options`：返回单据录入需要的人员、项目、商户、账户、币种、科目基础选项。
- `GET /api/document-entry/original-documents`：返回可冲正、可修正、可重录的原单据选项，可按单据类型过滤。

## 6. 操作者口径

正式口径：

- `createdBy` 保存创建人 `people.id`。
- 提交单据的 `actor` 保存提交人 `people.id`。
- 审核单据的 `reviewer` 保存审核人 `people.id`。
- 驳回单据的 `actor` 保存驳回人 `people.id`。
- `operatorPersonId` 是业务经办人，不等同于创建人或审核人。

当前阶段页面顶部提供“当前操作人”选择器。创建、提交、审核、驳回默认使用当前操作人；单据表单里的经办人、报销人、借款人仍按业务需要单独选择。

当前数据库的 `created_by`、`reviewed_by` 和审计日志 `actor` 字段仍可保持文本列，本阶段写入值统一为 `people.id`。后续权限阶段可以再增加外键或用户账户映射，不影响现有单据数据。

后续引入登录系统后，“当前操作人”由登录用户自动带出，并仍然映射到同一个人员主数据。

## 7. 单据类型字段

### 7.1 项目收入单

字段：

- 当前操作人
- 业务日期
- 经办人
- 项目
- 商户，按项目过滤
- 收入科目
- 收款账户
- 币种，默认跟随账户，可在账户支持时选择
- 原币金额
- USDT 折算金额
- 摘要

规则：

- 项目和商户必选。
- 商户必须属于所选项目。
- 收入科目必须是收入类科目。
- 收款账户必须是公司账户。

### 7.2 换汇单

字段：

- 当前操作人
- 业务日期
- 经办人
- 转出账户
- 转出币种
- 转出金额
- 转入账户
- 转入币种
- 转入金额
- USDT 成本
- 换汇科目
- 摘要

规则：

- 基础货币是 USDT。
- 当前过账模型以转入金额为 `amountMinor`，以 USDT 成本为 `usdtAmountMinor`。
- 转出账户作为对方账户保存。
- 转入账户生成资金批次。

### 7.3 账户划转单

字段：

- 当前操作人
- 业务日期
- 经办人
- 转出账户
- 转入账户
- 币种
- 金额
- 摘要

规则：

- 转出账户和转入账户不能相同。
- 币种必须与账户可用币种一致。
- 账户划转不确认收入或费用。

### 7.4 备用金发放单

字段：

- 当前操作人
- 业务日期
- 经办人
- 领取人
- 公司转出账户
- 人员备用金账户
- 币种
- 金额
- 摘要

规则：

- 领取人必选。
- 人员备用金账户按领取人过滤，只显示该人员名下 `petty_cash` 账户。
- 公司转出账户不能是人员备用金账户。

### 7.5 备用金退回单

字段：

- 当前操作人
- 业务日期
- 经办人
- 退回人
- 人员备用金账户
- 公司收回账户
- 币种
- 金额
- 摘要

规则：

- 退回人必选。
- 人员备用金账户按退回人过滤。
- 退回不确认费用。

### 7.6 备用金报销单

字段：

- 当前操作人
- 实际消费日期
- 报销人
- 项目
- 商户，可选，按项目过滤
- 费用科目
- 人员备用金账户
- 币种
- 原币金额
- 摘要

规则：

- 业务日期就是实际消费日期。
- 报销人必选。
- 费用科目必须影响费用报表。
- 备用金账户按报销人过滤。
- 备用金允许为负数，负数由过账和报表体现，不在录入时阻断。

### 7.7 借款发放单

字段：

- 当前操作人
- 业务日期
- 经办人
- 借款人
- 付款账户
- 币种
- 金额
- USDT 成本
- 借款科目
- 摘要

规则：

- 借款人必选。
- 借款不进入费用报表。
- 如果是非 USDT 借款，USDT 成本按 FIFO 或输入成本规则进入借款成本。

### 7.8 借款还款单

字段：

- 当前操作人
- 业务日期
- 经办人
- 借款人
- 收款账户
- 币种
- 金额
- 对应借款单，可选
- 摘要

规则：

- 借款人必选。
- 对应借款单未选择时，系统按借款人和币种冲减最早未还余额。

### 7.9 借款核销单

字段：

- 当前操作人
- 业务日期
- 经办人
- 借款人
- 核销科目
- 项目，可选
- 币种
- 金额
- 对应借款单，可选
- 摘要

规则：

- 借款人必选。
- 核销科目必须是费用或损失类。
- 核销不需要资金账户。

### 7.10 冲正、修正、重录

字段：

- 当前操作人
- 业务动作
- 原单据
- 原因摘要
- 其他字段按原单据类型显示

规则：

- 冲正和修正必须选择原单据。
- 原单据只允许从可操作的已审核单据中选择。
- 冲正单据类型必须与原单据类型一致。
- 当前阶段先支持冲正选择，不做复杂原单编辑器。

## 8. 页面结构

正式单据页拆成三个区域：

```mermaid
flowchart LR
  A["当前操作人"] --> B["单据列表"]
  A --> C["新建单据"]
  B --> D["提交 / 审核 / 驳回"]
  C --> E["按类型渲染表单"]
  E --> F["创建草稿"]
  F --> B
```

页面行为：

- 顶部选择当前操作人。
- 单据列表展示单据号、类型、日期、状态、经办人、项目、商户、摘要和操作按钮。
- 新建单据先选择单据类型，再显示对应字段。
- 字段控件使用下拉、分组下拉或可搜索选择，不暴露 ID 输入框。
- 页面仍保留草稿、提交、审核、驳回的最小闭环。

## 9. 选项联动

联动规则：

- 选择项目后，商户只显示该项目下商户。
- 选择人员后，备用金账户只显示该人员名下账户。
- 选择账户后，币种默认等于账户币种。
- 选择单据类型后，科目按 `category_type` 和报表影响字段过滤。
- 选择业务动作为冲正或修正后，原单据必选。
- 选择原单据后，单据类型锁定为原单据类型。

当联动导致已选择值不再合法时，前端清空该字段，并提示用户重新选择。

## 10. 错误处理

错误分三类：

| 类型 | 页面表现 |
| --- | --- |
| 主数据缺失 | 显示“请先到基础资料维护”，禁用创建按钮 |
| 字段不完整 | 字段下方显示中文校验信息，不提交请求 |
| 后端拒绝 | 保留表单内容，显示后端错误信息 |

前端校验只负责提升体验，最终合法性仍以后端服务为准。

## 11. 实施边界

为了降低风险，实施分两步：

1. 主数据选项能力：补齐人员、项目、商户、账户、科目和单据选项接口，并让前端能读取。
2. 单据录入重构：把单据页拆为当前操作人、单据列表、类型化表单和字段选择组件。

不在本阶段重做基础资料维护页的完整 CRUD。若缺少主数据，可先使用 seed 或后续单独阶段补基础资料管理。

## 12. 测试策略

测试覆盖：

- 主数据选项接口只返回启用数据。
- 商户按项目过滤。
- 备用金账户按人员过滤。
- 创建人、提交人、审核人、驳回人 payload 使用 `people.id`。
- 每种单据类型只生成自己的字段和 payload。
- 冲正动作要求原单据。
- 缺少必要主数据时禁用创建并显示中文提示。
- 现有单据创建、提交、审核、报表测试继续通过。

浏览器验证：

- 单据页不再出现“项目ID、商户ID、人员ID、账户ID”等手填字段。
- 选择项目后商户选项变化。
- 选择人员后备用金账户变化。
- 选择当前操作人后可创建、提交、审核、驳回单据。
